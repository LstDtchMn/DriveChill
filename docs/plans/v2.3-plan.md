## Confirmed Gaps From v2.2 Audit

### GAP-1 — API key auth bypasses RBAC write-block (Security — High)

**Problem:** `require_write_role` in `auth.py:282` explicitly returns early for API
key requests (`if api_key: return`). This means a viewer-role user can:
1. Call `POST /api/auth/api-keys` (session-authenticated, but no role check on
   that route) to create a new key.
2. Use that key to call write routes without any 403.

The `api_keys` table has no `created_by` or `role` column. `validate_api_key`
returns only `{id, name, key_prefix, scopes_json, ...}` — no role information.
The C# `ApiKeyService` has the same gap.

**Fix strategy (Phase C RBAC):**

*Migration `013_api_key_role.sql`:*
```sql
ALTER TABLE api_keys ADD COLUMN created_by TEXT;
ALTER TABLE api_keys ADD COLUMN role       TEXT NOT NULL DEFAULT 'admin';
```

*Python `auth_service.create_api_key`:*
- Accept `created_by_username: str` + `requesting_role: str`.
- If `requesting_role == "viewer"`, set `role = "viewer"` regardless of requested
  scopes (viewer keys cannot be promoted to admin).
- Store `created_by` and `role` in the row.

*Python `auth_service.validate_api_key`:*
- Return `role` in the result dict.

*Python `auth.py require_csrf` (write-block):*
- For `auth_type == "api_key"`, read `key_meta.get("role", "admin")`.
- If role is `"viewer"`, apply the same 403 block (except logout).
- This replaces the early-return shortcut.

*C# `ApiKeyService.CreateKeyAsync`:*
- Accept `string createdByUsername, string requestingRole`.
- Cap `role` to `"viewer"` if `requestingRole == "viewer"`.
- Persist `created_by` and `role`.

*C# `ApiKeyService.ValidateKeyAsync`:*
- Return `Role` in `ApiKeyMeta`.

*C# `Program.cs` auth middleware:*
- For API key auth: resolve `role` from `ApiKeyMeta.Role` (same capping logic
  as session: `viewer` → 403 on writes).

*Frontend `SettingsPage.tsx`:*
- `POST /api/auth/api-keys` already sends the session cookie; the backend will
  automatically cap the role. No frontend change needed.

**Tests:**
- Python: viewer creates key → key cannot write; admin creates key → key can write.
- C#: `ApiKeyServiceTests` — viewer-created key gets role=viewer; admin key role=admin.

### GAP-2 — Password change does not invalidate existing sessions (Security — High)

**Problem:** `PUT /api/auth/users/{id}/password` in `auth.py:266-282` updates
`password_hash` but does NOT delete the user's sessions. An attacker who has
compromised a session cookie retains access after the admin changes the password.
Same issue in `AuthController.cs:SetUserPasswordAsync`.

**Fix:**

*Python `auth.py change_user_password`:*
```python
await auth_service._db.execute(
    "DELETE FROM sessions WHERE user_id = ?", (user_id,)
)
await auth_service._db.commit()
```
Add this immediately after the `UPDATE users SET password_hash` commit.

*C# `AuthController.SetUserPassword`:*
```csharp
await _db.DeleteUserSessionsAsync(userId, ct);
```
`DeleteUserSessionsAsync` already exists (used by `DeleteUser`). Call it after
`SetUserPasswordAsync`.

**Tests:**
- Python `test_auth.py`: password change clears sessions; old session returns 401.
- C# `SessionServiceTests`: password change → validate returns null.

### GAP-3 — WebSocket session revalidation is message-count-based in Python, absent in C# (Medium)

**Problem:** Python `websocket.py:71-81` revalidates every 60 *messages*, which
is ≈ 60 s at 1 s polling but varies with poll interval and queue stalls. If the
user logs out from another tab the socket can continue for up to 60 messages.
C# `WebSocketHub` has no revalidation at all — a revoked/expired session keeps
the socket alive indefinitely.

**Fix:**

*Python `websocket.py`:*
- Replace `msg_count` integer with `last_revalidate: float = asyncio.get_event_loop().time()`.
- In the loop, after getting a snapshot, check
  `if time.monotonic() - last_revalidate >= 60.0` and revalidate then reset.
- Remove `_SESSION_REVALIDATE_INTERVAL` constant.

*C# `WebSocketHub.HandleAsync`:*
- After `websocket.AcceptAsync()`, extract the session token from the `Cookie`
  header (same parsing as `Program.cs` middleware).
- Maintain `DateTimeOffset lastRevalidate = DateTimeOffset.UtcNow`.
- In the broadcast loop: if `(DateTimeOffset.UtcNow - lastRevalidate).TotalSeconds >= 60`,
  call `await _sessions.ValidateSessionAsync(token, ct)`. If null, close with
  `WebSocketCloseStatus.PolicyViolation` and return.

**Tests:**
- Python: mock `validate_session` to return None after first call; assert socket
  closes within one revalidation window.
- C#: time-advance mock; assert `ValidateSessionAsync` is called after 60 s.

### GAP-4 — C# drive provider is not abstracted (Medium — blocks USB support)

**Problem:** `DriveMonitorService.cs` (736 lines) directly shells out to
`smartctl`. There is no interface, making it impossible to:
- Swap in a mock for unit tests (all drive tests currently require smartctl).
- Add a second provider (USB HID fan controllers like Corsair/NZXT).

**Design:**

```csharp
/// <summary>Abstraction over a physical drive data source.</summary>
public interface IDriveProvider
{
    Task<IReadOnlyList<DriveInfo>> DiscoverDrivesAsync(CancellationToken ct);
    Task<DriveHealthSnapshot?> GetHealthAsync(string devicePath, CancellationToken ct);
    Task<SelfTestResult> StartSelfTestAsync(string devicePath, string type, CancellationToken ct);
    Task<SelfTestStatus> GetSelfTestStatusAsync(string devicePath, CancellationToken ct);
}
```

*`SmartctlDriveProvider`* — move all `smartctl` shelling logic out of
`DriveMonitorService` into this class. `DriveMonitorService` calls
`IDriveProvider` only.

*`MockDriveProvider`* — returns configurable fake drives/snapshots; registered
in DI when `DRIVECHILL_HARDWARE_BACKEND=mock`.

*`DriveMonitorService`* — constructor takes `IDriveProvider`; all direct
process-spawn calls removed.

**DI registration in `Program.cs`:**
```csharp
if (settings.HardwareBackend == "mock")
    builder.Services.AddSingleton<IDriveProvider, MockDriveProvider>();
else
    builder.Services.AddSingleton<IDriveProvider, SmartctlDriveProvider>();
builder.Services.AddSingleton<DriveMonitorService>();
```

**Tests unlocked:** `DriveMonitorServiceTests` can inject `MockDriveProvider`
and assert temperature injection, health badge logic, self-test state machine.
Target: 10+ new C# tests.

## New Features — v2.3

### 5. Prometheus Metrics (C# backend)

**Why:** Operators running DriveChill in homelab/Docker setups commonly scrape
metrics with Prometheus + Grafana. The Python backend has no metrics endpoint
either — both should be added together.

**Dependency:** `prometheus-net.AspNetCore` (C#); `prometheus-client` (Python).

**Metrics to expose (`/metrics`):**

| Metric | Type | Labels |
|--------|------|--------|
| `drivechill_sensor_poll_duration_seconds` | Histogram | backend |
| `drivechill_sensor_readings_total` | Counter | sensor_type |
| `drivechill_alert_events_total` | Counter | rule_id, severity |
| `drivechill_fan_speed_pct` | Gauge | fan_id |
| `drivechill_drive_temp_celsius` | Gauge | drive_id |
| `drivechill_websocket_connections_active` | Gauge | — |
| `drivechill_webhook_deliveries_total` | Counter | success (true/false) |

**C# implementation:**
- Add `prometheus-net.AspNetCore` NuGet.
- `app.UseMetricServer()` in `Program.cs` (gated behind `AppSettings.EnableMetrics`).
- In `SensorWorker`: wrap `PollAsync` with `Histogram.Observe`; update fan/temp
  gauges from `last_applied_speeds` and sensor snapshot.
- In `WebSocketHub`: increment/decrement `Gauge` on connection open/close.
- In `WebhookService`: increment counter on each delivery attempt.

**Python implementation (`backend/app/api/routes/metrics.py`):**
- `GET /metrics` using `prometheus_client.generate_latest()`.
- Same metric names for cross-backend scraping compatibility.
- Register collector in `main.py`; gated by `DRIVECHILL_ENABLE_METRICS=true`.

**Auth:** `/metrics` is exempt from session auth (Prometheus scrapers use
network-level isolation, not cookies). Document this in the route.

### 6. API Key Scopes UI

**Why:** The frontend Settings page shows API keys but not their scopes. Users
cannot see what a key can do or create a read-only key through the UI.

**Current contract:**
```
POST /api/auth/api-keys  ← { name, scopes: ["sensors:read", ...] }
GET  /api/auth/api-keys  → [{ id, name, key_prefix, scopes_json, created_at, ... }]
```

**Frontend changes (`SettingsPage.tsx`):**
- API key create form: add multi-select or checkbox list for scopes, grouped by
  domain (sensors, fans, profiles, alerts, webhooks, temperature_targets, analytics).
- API key list: expand each row to show scope badges.
- Show the key's effective `role` badge (admin / viewer) next to the name.

**No backend changes needed** once GAP-1 is fixed (backend already enforces
role ceiling; scopes already stored).

## Milestone Summary

| Milestone | Items | Outcome |
|-----------|-------|---------|
| **v2.3.0-rc1** | GAP-1 (API key role ceiling), GAP-2 (password change invalidates sessions) | RBAC fully watertight — no session or key escalation paths |
| **v2.3.0-rc2** | GAP-3 (WebSocket time-based revalidation, C# + Python) | Long-lived sockets correctly expire with sessions |
| **v2.3.0** | GAP-4 (C# drive provider abstraction + MockDriveProvider + tests) | DriveMonitorService testable; unlocks USB hardware |
| **v2.3.1** | Prometheus metrics (C# + Python) | Operator observability |
| **v2.3.2** | API key scopes UI | Self-serve key management |

**Rationale for ordering:**
- Security gaps (GAP-1, GAP-2) ship first — they close real privilege-escalation
  paths introduced by the v2.2 RBAC feature.
- GAP-3 (WebSocket) is correctness, not an escalation path — slightly lower
  urgency but should land before v2.3.0 final.
- Drive abstraction (GAP-4) is architectural prep; safe to land after security
  fixes stabilise.
- Prometheus and scopes UI are value-add and have no blocking dependencies.

## Known Deferred Items (not targeted for v2.3)

- USB HID fan controller support (Corsair Lighting Node, NZXT Smart Device 2)
  — requires GAP-4 drive abstraction first; also needs separate `IFanProvider`
  abstraction in `FanService`.
- Session token rotation on privilege operations (role change, password change
  on own account) — tokens are short-lived (8 h) so risk is low after GAP-2.
- WebSocket auth renegotiation on role-change events — would require a server-push
  "your role changed, reconnect" message; low priority.
- Python drive provider abstraction — Python uses subprocess + smartctl already
  structured as a service class; lower coupling than C# so less urgent.
