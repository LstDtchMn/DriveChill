### Profile Export / Import

**Current Python contract** (must match exactly):

```
GET /api/profiles/{id}/export ‚Üí {
  "export_version": 1,             // NOT "schema_version"
  "profile": {
    "name": str,
    "preset": str,
    "curves": FanCurve[]           // original IDs preserved in export
  }
}

POST /api/profiles/import ‚Üź {     // flat profile body, NOT wrapped envelope
  "name": str | null,
  "preset": str,
  "curves": FanCurve[]
}
‚Üí { "success": true, "profile": Profile }
```

The frontend (`api.ts:193-199`) sends the flat `{name, preset, curves}` object directly
to `/api/profiles/import` ‚ÄĒ no `export_version` envelope.

**C# implementation:**
- `GET /api/profiles/{id}/export`: serialize from `SettingsStore.LoadProfiles()`, strip
  `id` and `is_active` fields, return `{ export_version: 1, profile }`.
- `POST /api/profiles/import`: accept `ImportProfileRequest { Name?, Preset, Curves[] }`,
  generate fresh profile ID + fresh curve IDs via `RandomNumberGenerator`, append to
  `_store`, return the new profile.
- Do NOT generate a wrapped envelope or require `schema_version` in the import body.

**Implementation:**

## New Features ‚ÄĒ v2.2

### 1. PID Fan Controller for Temperature Targets

**Why:** The current proportional controller can oscillate when the drive temperature
has a slow feedback loop (e.g., NVMe under sustained load overshoots the tolerance band
before the fan speed change takes effect).

**Control loop integration:**
- `TemperatureTargetService.Evaluate` is called once per `SensorWorker` tick
  (`SensorWorker.cs:137` ‚ÄĒ `_store.PollIntervalMs`, typically 1000 ms).
- Each target maintains its own `PidState` instance.

**DB migration `011_pid_fields.sql`:**
```sql
ALTER TABLE temperature_targets ADD COLUMN kp REAL NOT NULL DEFAULT 1.0;
ALTER TABLE temperature_targets ADD COLUMN ki REAL NOT NULL DEFAULT 0.0;
ALTER TABLE temperature_targets ADD COLUMN kd REAL NOT NULL DEFAULT 0.0;
```

**PidState struct (per target, in-memory only):**
```
struct PidState {
    double Integral;       // accumulated integral term
    double PreviousError;  // error(t-1) for derivative
    long   LastTickMs;     // DateTimeOffset.UtcNow.ToUnixTimeMilliseconds()
}
```

**dt computation:**
- `dt = (nowMs - state.LastTickMs) / 1000.0` (seconds, float).
- On first evaluation or after target toggle/update, `LastTickMs == 0` ‚Üí skip integral
  and derivative terms for one tick (warm-up), use proportional-only.
- Guard: `if dt <= 0 || dt > 30` ‚Üí treat as proportional-only (prevents integral wind-up
  from stale state after long pauses like sleep/hibernate).

**State reset conditions:**
- Target is toggled off and back on (`SetEnabledAsync`).
- Target fields are updated (`UpdateAsync`).
- Application restart (PidState is in-memory only, not persisted).
- State is keyed by target ID; removed when target is deleted.

**Derivative filtering:**
- Apply a simple low-pass EMA to the derivative term to suppress sensor noise:
  `filteredDerivative = 0.7 * rawDerivative + 0.3 * state.PreviousFilteredDerivative`
- Add `PreviousFilteredDerivative` to `PidState`.

**Output saturation:**
- Raw PID output is clamped to `[minFanSpeed, 100]`.
- Anti-windup: when output is clamped, stop accumulating integral in the clamped direction
  (conditional integration). Specifically: if output >= 100 and error > 0, don't add to
  integral; if output <= minFanSpeed and error < 0, don't subtract from integral.

**Default gains:** `kp = 1.0, ki = 0.0, kd = 0.0` ‚Üí pure proportional, fully
backwards-compatible. Users opt into PID by setting `ki`/`kd` > 0.

**Frontend (`TemperatureTargetsPage.tsx`):**
- Add collapsible "Advanced: PID gains" section to `TargetForm`.
- Show `kp`, `ki`, `kd` number inputs with tooltips explaining each term.
- Only visible when the user clicks "Show advanced".

**Files:** `temperature_target_service.py`, `TemperatureTargetService.cs`,
`temperature_targets.py`, `TemperatureTargetsController.cs`, migration
`011_pid_fields.sql`, `TemperatureTargetsPage.tsx`, `types.ts`

### 2. Multi-User RBAC

**Why:** Currently a single `admin` user. Adding a `viewer` role enables read-only
monitoring dashboards, home-assistant integrations, and shared access without giving
control privileges.

**Current gaps that must be addressed:**

1. **Session payloads carry no role.** Python `auth.py:126` stores
   `{"auth_type": "session", "session": session}` where session is
   `{token, csrf_token, username, ip, user_agent, created_at, expires_at}` ‚ÄĒ no role
   field. C# `AuthController:45` returns `{username}` only.

2. **API keys are not owner-bound.** Python `auth_service.py:157` and C#
   `ApiKeyService.cs:48` create keys without associating a `created_by` user or
   inheriting a role ceiling.

**Design ‚ÄĒ phased:**

**Phase A: Data model + session propagation**
- Migration: `ALTER TABLE users ADD COLUMN role TEXT NOT NULL DEFAULT 'admin'`.
  Existing users get `admin`.
- `SessionService.LoginAsync` / Python `auth_service.create_session`: write `role`
  into the sessions table (new column) and return it in the session dict.
- `auth.py:126` session dict and C# auth middleware: include `role` in the
  `request.state.auth_info` / `HttpContext.Items`.

**Phase B: Write-route gating**
- Define a `WRITE_ROUTES` set: all `POST/PUT/DELETE/PATCH` routes *except*
  `/api/auth/login`, `/api/auth/logout`, `/api/auth/session`.
- Python `require_auth()` in `auth.py`: after validating the session, if `role == "viewer"`
  and the request method + path matches a write route, return 403
  `{"detail": "Viewer role cannot perform write operations"}`.
- C# `Program.cs` auth middleware: same check using `HttpContext.Items["role"]`.
- This is a blanket deny ‚ÄĒ simpler and more secure than per-route annotation, because
  new write endpoints are automatically blocked for viewers.

**Phase C: API key role ceiling**
- `create_api_key`: accept optional `max_role` parameter. If the creating user is
  `viewer`, the key's effective scope is capped to `read:*` regardless of requested
  scopes. Store `created_by_username` and `max_role` in `api_keys` table.
- On API key auth, resolve the effective role as `min(key.max_role, creator.role)`.
  If the creator's role has since been downgraded, the key is also downgraded.

**Phase D: Frontend**
- Settings page: "User Management" section ‚ÄĒ list users (username, role, created_at),
  add user form (username, password, role dropdown), delete user button.
- Conditionally hide write controls (create profile, add alert, etc.) when
  `authStore.role === 'viewer'`.

**Files:** migration `012_rbac.sql`, `auth_service.py`, `auth.py`,
`SessionService.cs`, `DbService.cs`, `AuthController.cs`, `Program.cs`,
`ApiKeyService.cs`, `SettingsPage.tsx`, `authStore.ts`, `types.ts`

### 5. E2E Tests in CI

**Why:** Analytics and TemperatureTargets pages have no Playwright coverage, and the
existing 5 specs don't run in GitHub Actions.

**Current Playwright config issue:** `playwright.config.ts:41-44` starts the Next.js
dev server via `webServer`, but it points at `http://localhost:8085` for the API
backend. In CI, no backend is running on `:8085` ‚ÄĒ tests will fail on every API call.

**Design:**

The workflow must orchestrate three processes:

```yaml
# .github/workflows/e2e.yml
jobs:
  e2e:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@<sha>
      - uses: actions/setup-python@<sha>
        with: { python-version: '3.12' }
      - uses: actions/setup-node@<sha>
        with: { node-version: '20' }

      # 1. Install backend deps + start mock backend in background
      - run: pip install -r backend/requirements.txt
      - run: |
          cd backend
          DRIVECHILL_HARDWARE_BACKEND=mock \
          DRIVECHILL_HOST=127.0.0.1 \
          DRIVECHILL_PORT=8085 \
          python drivechill.py --headless &
      - run: |
          # Wait for backend to be ready (max 30s)
          for i in $(seq 1 30); do
            curl -sf http://127.0.0.1:8085/api/health && break
            sleep 1
          done

      # 2. Install frontend deps + Playwright browsers
      - run: cd frontend && npm ci
      - run: cd frontend && npx playwright install --with-deps chromium

      # 3. Run Playwright (Next.js dev server is started by webServer config)
      - run: cd frontend && npm run test:e2e
        env:
          NEXT_PUBLIC_API_URL: http://127.0.0.1:8085
          NEXT_PUBLIC_WS_URL: ws://127.0.0.1:8085/api/ws
```

**New test specs:**
- `tests/e2e/analytics.spec.ts` ‚ÄĒ time window selector, sensor filter, stat cards render
- `tests/e2e/temperature-targets.spec.ts` ‚ÄĒ create target, toggle enabled, delete,
  relationship map renders

**Files:** `.github/workflows/e2e.yml`, `frontend/tests/e2e/analytics.spec.ts`,
`frontend/tests/e2e/temperature-targets.spec.ts`

## Milestone Summary (revised ‚ÄĒ parity and tests BEFORE new features)

| Milestone | Items | Outcome |
|-----------|-------|---------|
| **v2.2.0-rc1** | C# profile export/import, auth_log in C#, C# test expansion (target: 60+ tests) | Full C# parity; regression safety net |
| **v2.2.0-rc2** | ¬įF gauge fix, custom modals, E2E in CI | UX polish + CI green |
| **v2.2.0** | PID controller (both backends) | Smarter temperature control |
| **v2.2.1** | RBAC (phases A-D) | Multi-user access |
| **v2.3+** | C# drive provider abstraction, USB controller support (Corsair/NZXT HID) | Hardware breadth |

**Rationale for ordering:**
- Parity and tests come first (rc1) so that PID and RBAC changes in later milestones
  have a regression safety net.
- ¬įF gauge and custom modals (rc2) are small, low-risk, and unblock E2E tests (confirm
  dialogs block Playwright by default).
- PID is isolated to temperature targets ‚ÄĒ no auth/session changes.
- RBAC is the highest-risk item (touches auth middleware in both backends, session format,
  API key model). It goes last so all other changes are stable before the auth layer is
  modified.
