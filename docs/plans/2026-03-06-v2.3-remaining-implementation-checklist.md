## 3. Gate Before Starting Release 2

Do not start new feature work until all of the following are true:

- the current dirty working tree is reviewed and intentionally preserved
- Python, C#, and frontend baselines stay green after any doc or test changes
- release-1 parity work is covered by direct tests, not only by incidental
  route coverage
- the Windows-local Playwright path is fixed so local E2E work is not Linux-only
- docs are updated so implementation agents are not following stale plans
- phases 1-4 are treated as strict gates, not parallel background work

Reason: the current branch contains substantial new behavior. Building more
features on top of code that is still under-documented and under-tested will
create avoidable regressions.

Implement in this order:

1. Stabilize the current v2.3 branch
2. Add missing direct tests for release-1 parity work

load-input model is stable. Those later features depend on clean capability and
sensor-source semantics.

## 10. Phase 6: Load-Based Fan-Curve Inputs — ✅ COMPLETE (2026-03-06)

### 10.1 Scope decision

Do not redesign the public fan-curve schema in this phase.

Keep the existing point shape and treat the existing numeric x-axis value as the
curve input value regardless of whether the source is temperature or load.
This avoids an unnecessary route/schema version bump.

Initial supported non-temperature inputs:
- `cpu_load`
- `gpu_load`

`cpu_power` should only be added if both backends can expose it reliably. It is
not a gate for shipping CPU/GPU load curves.

What is missing is control-path support and clear metadata.
