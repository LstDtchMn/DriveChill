## CRITICAL (block deployment)

### C1. SSRF — No private IP/hostname blocklist on URL inputs

**Affects:** Webhook `target_url` (webhooks.py:19-27), Machine `base_url` (machines.py:22-28)

Both validators only check `http://` or `https://` prefix. Users can point at `http://169.254.169.254/` (cloud IMDS), `http://localhost:8085/api/...` (self), or any LAN target. The webhook client also has `follow_redirects=True` (webhook_service.py:25), which lets an external redirect bounce to internal targets, bypassing even a future blocklist. The machine monitor client also follows redirects.

**Fix:** Add `ipaddress`-based blocklist to both validators. Set `follow_redirects=False` on both HTTP clients.

---

### C2. `POST /api-keys` and `DELETE /api-keys/{key_id}` missing `require_auth`

**File:** auth.py routes

These routes are on the `auth_router`, which is included **without** `_auth_deps` (because login/setup must be public). They only have `Depends(require_csrf)`. On localhost deployments (`auth_required=False`), `require_csrf` is a no-op — anyone can create or revoke API keys without authentication.

**Fix:** Add `Depends(require_auth)` to both endpoints.

---

### C3. Plaintext API key stored in `machines` table, returned via `GET /api/machines`

**Files:** 005_machine_registry.sql, machine_repo.py, machines.py:76-81

The `machines.api_key` column stores credentials in plaintext (violates the design doc which says "no plaintext key storage"). `_decorate_machine` spreads `**machine` into the API response, so any authenticated browser session receives the raw key for every registered machine. A DB dump or XSS leaks live credentials.

**Fix:** Encrypt at rest or store only `api_key_id` (FK to `api_keys`). Redact `api_key` from all GET responses.

---

### C4. Webhook signing secret returned in plaintext from `GET /api/webhooks`

**File:** webhook_service.py:52, webhooks.py:33-34

`get_config()` returns the raw `signing_secret`. Any session/API-key holder can read it. The secret should be write-only from the API consumer's perspective.

**Fix:** Return `has_signing_secret: bool` instead of the raw value.

---

### C5. Prometheus label injection via sensor ID

**File:** main.py:357-361

The sanitization (`replace('"', "").replace("\\", "")`) misses `}`, `{`, and newlines. A sensor ID with a newline can inject arbitrary metric lines. A `}` can close the label block early and corrupt parsing.

**Fix:** Whitelist-sanitize: `re.sub(r'[^a-zA-Z0-9_.\-]', '_', r.id)`

---

### C6. Docker `docker-compose.yml` mounts `/dev:/dev` and runs as root

**Files:** docker-compose.yml:22, Dockerfile

Full `/dev` mount + `SYS_RAWIO` + root user = a compromised container can read any disk on the host. Labeled "preferred minimal permissions profile" which is misleading. No `.dockerignore` exists, so DB files, `node_modules`, and `.env.local` are included in the build context.

**Fix:** Mount only specific devices needed, add `USER` to Dockerfile, create `.dockerignore`.

---

### C7. Internal token comparison uses `==` (timing attack)

**File:** auth.py dependencies:20

`token == settings.internal_token` is subject to short-circuit evaluation. Should use `hmac.compare_digest()`.

## HIGH (fix before GA)

### H1. `validate_api_key` called twice per mutating request

**File:** auth.py dependencies:51-61, 122-128

Both `require_auth` and `require_csrf` independently call `validate_api_key()`, each doing a SELECT + UPDATE `last_used_at`. Doubles DB write load on every POST/DELETE with API key auth.

**Fix:** Cache the auth result in `request.state.auth_info` from `require_auth`; have `require_csrf` check that.

---

### H2. API key scopes stored but never enforced

**File:** auth_service.py

`scopes_json` is stored and returned but no route checks scopes. Every API key has full access regardless of declared scopes. Arbitrary scope strings are accepted with no validation.

**Fix:** Either enforce scopes in the dependency chain or remove the field from the API surface to prevent false confidence.

---

### H3. Webhook delivery log grows without bound

**File:** 006_api_keys_webhooks.sql

`webhook_delivery_log` has no pruning job (unlike `sensor_log` and `auth_log`). High-frequency alerts + retries = thousands of rows quickly.

**Fix:** Add a `prune_delivery_log()` method and wire it into the existing periodic cleanup loop.

---

### H4. Exponential backoff unbounded — up to 4+ hours per retry

**File:** webhook_service.py:186

`backoff * (2 ** (attempt - 1))` with `max_retries=10` and `backoff=30s` = 15,360s on attempt 10. Tasks accumulate holding the event loop.

**Fix:** Cap: `min(backoff * (2 ** (attempt - 1)), 60.0)`

---

### H5. Windows service script points to tray app entry point

**File:** install_windows_service.ps1:13

`drivechill.py` is the tray entry point. A Windows Service runs in Session 0 with no desktop — pystray will crash immediately. Should use `uvicorn app.main:app` for headless mode.

---

### H6. CSRF token comparison not constant-time

**File:** auth.py dependencies:138

`x_csrf_token != session["csrf_token"]` uses `!=`. Lower severity than C7 (requires existing session) but should use `hmac.compare_digest()` consistently.

---

### H7. `_classify_failure` substring match causes false `auth_error`

**File:** machine_monitor_service.py:175

`"401" in msg` matches timeout messages like "timed out after 401ms". Should only trust `httpx.HTTPStatusError` for status codes.

## NEW FINDINGS FROM REMEDIATION

### R1. DNS rebinding bypasses SSRF validation (Critical)

**Confidence: 95**
**File:** url_security.py (validation at config-save time only)

`validate_outbound_url` resolves the hostname via `socket.getaddrinfo` once at config-save time. `httpx` re-resolves DNS at every request time. An attacker controls a domain, passes validation with a public IP, then flips the DNS record to `169.254.169.254` (IMDS) or `127.0.0.1` with a near-zero TTL. Every subsequent poll/webhook delivery hits the internal target.

**Fix:** Re-validate the resolved IP immediately before each outbound request in `_fetch_remote` and `_dispatch_payload`:
```python
ok, reason = validate_outbound_url(target_url)
if not ok:
    raise RuntimeError(f"URL blocked at request time: {reason}")
```

Deeper fix: resolve hostname, validate IP, then make the HTTP request to the IP directly with a `Host:` header.

### R2. Python Prometheus regex admits literal backslash (Critical)

**Confidence: 100**
**File:** main.py:357

```python
sid = re.sub(r"[^a-zA-Z0-9_.\\-]", "_", str(r.id))
```

In a raw string, `\\-` is two characters: literal backslash + hyphen. The backslash is now an *allowed* character in the sanitized label. A backslash in a Prometheus label value is an escape character and can corrupt scrape output. The C# equivalent is correct (`\-` is just an escaped hyphen in C# regex).

**Fix:**
```python
sid = re.sub(r"[^a-zA-Z0-9_.-]", "_", str(r.id))
```

### R3. Machine `api_key` silently NULLed on partial PUT (Critical)

**Confidence: 95**
**File:** machines.py routes, machine_repo.py

When `PUT /api/machines/{id}` omits `api_key` (e.g., user only changes `name`), Pydantic sets `body.api_key = None`. The route passes `api_key=None` to `repo.update()`, which writes `api_key = NULL` to the DB. The stored credential is silently erased, breaking auth to that machine.

The webhook remediation correctly solves this same problem with `_KEEP_SECRET` sentinel + `model_fields_set` check. The machine route does not have the equivalent guard.

**Fix:** Apply the same `_KEEP_SECRET` / `model_fields_set` pattern used in the webhook update path.

---

### R4. Session double-validation not fixed by H1 cache (High)

**Confidence: 90**
**File:** auth.py dependencies:121-123

The `request.state.auth_info` cache early-return in `require_csrf` only checks for `auth_type == "api_key"`. When session auth is used on endpoints with both `Depends(require_auth)` and `Depends(require_csrf)`, `validate_session` is called twice — once in each dependency — triggering two DB writes (`last_active` + `expires_at`) per mutating request. The `POST /api-keys` and `DELETE /api-keys/{key_id}` endpoints are affected for browser-session users.

**Fix:** Extend the cache check to also branch on `auth_type == "session"`, using the cached session's `csrf_token` for the CSRF check instead of re-querying:
```python
existing = getattr(request.state, "auth_info", None)
if existing:
    if existing.get("auth_type") == "api_key":
        return
    if existing.get("auth_type") == "session":
        csrf_token = existing["session"]["csrf_token"]
        if not x_csrf_token or not hmac.compare_digest(str(x_csrf_token), str(csrf_token)):
            raise HTTPException(status_code=403, detail="CSRF token invalid or missing")
        return
```

### R5. IPv4-mapped loopback bypass on Python < 3.11 (High)

**Confidence: 83**
**File:** url_security.py

`http://[::ffff:127.0.0.1]/` — on Python 3.10, `IPv6Address("::ffff:127.0.0.1").is_loopback` returns `False`. The IPv4-mapped `127.x.x.x` address passes all checks and reaches loopback. Similarly, `fc00::/7` (ULA) is not caught by `is_private` on Python 3.10.

**Fix:**
```python
if isinstance(ip, ipaddress.IPv6Address) and ip.ipv4_mapped is not None:
    mapped = ip.ipv4_mapped
    if mapped.is_loopback or mapped.is_link_local or (mapped.is_private and not allow_private):
        return False, "IPv4-mapped address targets are not allowed"
```

Or pin minimum Python version to 3.11.

### R6. `_KEEP_SECRET` sentinel defined after first use (High)

**Confidence: 90**
**File:** webhook_service.py (line ~248, used at line ~102)

`_KEEP_SECRET` is used in `update_config` but defined at module bottom. Works at runtime (Python resolves module globals at call time), but any import-time call to `update_config` would raise `NameError`. A linter/type-checker will flag this.

**Fix:** Move `_KEEP_SECRET = object()` to the top of the module, after the `logger` assignment.

### R7. `prune_delivery_log` COUNT+DELETE is non-atomic (Medium)

**Confidence: 85**
**File:** webhook_service.py

The method first does `SELECT COUNT(*)`, then a separate `DELETE ... LIMIT ?`. A row inserted between the two makes the `to_delete` count stale. Also, `cursor.rowcount` for `DELETE WHERE id IN (SELECT ...)` is unreliable on older SQLite (< 3.35) and may return `-1`.

**Fix:** Single atomic statement:
```python
cursor = await self._db.execute(
    "DELETE FROM webhook_delivery_log WHERE id NOT IN "
    "(SELECT id FROM webhook_delivery_log ORDER BY id DESC LIMIT ?)",
    (safe_max,),
)
```

### R8. `docker-compose.privileged.yml` nullifies non-root fix (Medium)

**Confidence: 85**
**File:** docker-compose.privileged.yml:18

`privileged: true` re-grants the container root namespace access regardless of the in-container UID, nullifying the `USER drivechill` fix. No warning or guidance on which specific capabilities (`CAP_SYS_RAWIO`) would suffice instead.

**Fix:** Add a warning comment and document a capabilities-only alternative:
```yaml
# WARNING: privileged mode overrides non-root USER and grants full host device access.
# Prefer using specific capabilities instead:
# cap_add: [SYS_RAWIO]
# devices: [/dev/port:/dev/port]
```

### R9. Windows service `cmd /c` binPath is not valid for SCM (Medium)

**Confidence: 88**
**File:** install_windows_service.ps1:13

The `binPath` uses `cmd /c ... uvicorn`. SCM expects a Win32 service executable that calls `StartServiceCtrlDispatcher`. `cmd.exe` is not a service executable — the service will show as "Starting" and time out. The fix changed *what* is executed but not *how* it is registered.

**Fix:** Use NSSM (`nssm install DriveChill ...`) or `pywin32`'s service wrapper.

### R10. `last_error` can leak bearer token via httpx exception messages (Medium)

**Confidence: 82**
**File:** machine_monitor_service.py, machines.py

`last_error` is stored as `str(exc)[:300]` and returned unredacted in `GET /api/machines`. `httpx` exceptions can embed request URLs or other context. While `_public_machine()` strips `api_key`, `last_error` passes through as-is.

**Fix:** Apply `_redact_url_for_log()` (or equivalent) to `last_error` before storage.

### R11. Webhook test does not verify HMAC digest value (Low)

**Confidence: 82**
**File:** test_webhooks.py

The assertion checks `signatures[0].startswith("sha256=")` but never computes the expected HMAC and compares. A bug producing wrong digests would pass the test.

**Fix:** Independently compute expected HMAC from captured request body and assert equality.

### R12. Dockerfile `chown -R /app` creates oversized layers (Low)

**Confidence: 80**
**File:** Dockerfile:29-31

`chown -R /app` after `COPY` creates a duplicate layer with ownership metadata. Standard pattern is `COPY --chown=drivechill:drivechill` on each `COPY` instruction.
