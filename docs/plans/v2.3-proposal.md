## Part 2 — Competitor Landscape

### FanControl (Windows, open-source, NickBoon)
**What they do better:**
- Virtual sensor system is fully first-class: Mix (weighted blend), Plus/Minus (offsets), Clamp, Truncate, Moving Average — each composable into the next
- Fan stop (zero-RPM) detection baked into calibration; result auto-populates curve floor
- Plugin ecosystem: hardware backends (Corsair, NZXT, Gigabyte, ASUS, LiquidCooler) are dlls dropped into a folder — no fork needed
- Trigger-based profile switching built in ("switch to profile X when temperature sensor Y exceeds Z")
- Sensor graph overlay — live sparkline per sensor composited on any page

**DriveChill advantages over FanControl:**
- Headless/server mode (no GUI required)
- Drive SMART monitoring built-in
- REST API with auth and scopes — usable from CI/scripts
- Multi-machine dashboard
- Web Push + email alerts

### CoolerControl (Linux, open-source)
**What they do better:**
- liquidctl device abstraction — NZXT, Corsair Commander, Aquacomputer, and others work day-one
- System daemon with udev rules — survives across reboots without user session
- Fan curve driven by CPU frequency (load-based), not just temperature
- Profile groups: multiple concurrent profiles in separate "override" channels
- Fan curve function types: Linear, Step, Fixed, Graph — not just linear interpolation
- Lighting control unified with fan control per device

**DriveChill advantages:**
- SMART drive monitoring + health trending
- Web UI accessible remotely
- Alert pipeline with webhooks
- Multi-machine support
- Windows native support

### Argus Monitor (Windows, commercial, $40)
**What they do better:**
- GPU load-based fan curves (not just GPU temp — actual utilization %)
- Battery health monitoring (laptops)
- S.M.A.R.T. with visual failure prediction charts (sector reallocation trending over time)
- Configurable tray overlays with any sensor
- CrystalDiskInfo integration

**DriveChill advantages:**
- Free and open source
- Remote/headless capable
- REST API
- Push notifications

### HWiNFO (monitoring only, reference point)
- Richest sensor set: VRM temp, PCIe bandwidth, CPU package power, per-core freq
- Remote Sensor endpoint (localhost HTTP) — popular for third-party integrations
- Shared memory interface used by countless overlay tools

**Takeaway:** HWiNFO exposes power/load metrics DriveChill doesn't currently use as curve inputs. Users with HWiNFO running could feed those into DriveChill via virtual sensor bridging.

### iCUE / CAM / Armoury Crate (closed-source OEM)
These are ecosystem lock-in tools. Their value is RGB sync + deep proprietary hardware access. Not worth emulating architecture-wise, but their **trigger-chain** model (sensor condition → action → notification) is worth studying.

## Part 3 — Critique of the Gemini/Codex Proposal

### Error 1: "PID control is a Release 2 feature"
**Wrong.** PID is already fully implemented in Python (`temperature_target_service.py:52`, complete with integral anti-windup and derivative term). The C# port is the only thing missing — proportional is there, PID is not. The Gemini proposal implies PID needs to be designed from scratch; it only needs to be (a) ported to C# and (b) exposed in the UI. The `TemperatureTargetsPage.tsx` currently shows *only* `clientProportional` and has no PID configuration UI at all.

**Correction:** PID C# port + PID UI is a Release 1 gap, not Release 2.

### Error 2: "Composite sensor support needs to be designed"
**Partially wrong.** The `FanCurve` model already has `sensor_ids: list[str]` and `resolve_composite_temp()` implements MAX fallback. The gap is:
- C# does not support `sensor_ids` at all
- Python only supports MAX; MIN/AVG/WEIGHTED do not exist
- There is no UI to set composite sensor lists on a curve

**Correction:** This is a C# port + model extension + UI task, not a greenfield design.

### Error 3: "Fan calibration is missing"
**Wrong framing.** `FanTestPanel.tsx` has a complete benchmark flow (stall detection, max RPM sweep, min-speed detection). The gap is that calibration results are not auto-applied. After a benchmark, the user must manually set `min_speed_pct` and `zero_rpm_capable`. Adding "Apply calibration" as one button click that calls PUT `/api/fans/{id}/settings` with the measured values is a small, targeted addition.

### Error 4: "No fail-safe"
**Misleading.** Both backends have `ReleaseFanControl()` / `release_fan_control()` which hands fans back to BIOS. Python also has `_sensorPanic` / `_tempPanic` modes. C# has the same. The Gemini proposal correctly reframes this as "hard-crash hardware-memory fallback" but then proposes no concrete implementation. Real crash resilience for a PC fan controller means:
- A watchdog that monitors the process externally (Windows Service / systemd) — already partly covered by the install scripts
- A startup safety profile applied at process boot before any curve evaluation begins (not currently implemented)
- Documentation of what happens on power-loss (BIOS takes over — actually fine by design)

The concrete gap is **startup safety profile** (first N seconds of operation: run at a safe fixed speed before curve data is loaded).

### Error 5: "Quiet Hours UI is a new feature"
**Wrong.** The backend CRUD (`quiet_hours.py`) is complete: create/update/delete by day-of-week, HH:MM start/end, profile_id. The frontend has no page at all for it. This is a **UI-only task**, not a backend feature.

### Error 6: "API key scope UI is a Release 1 item"
**Already shipped.** The scope picker was implemented and the domain list extended to all 13 domains in the current session (v2.3 work). This should be removed from the roadmap.

### Error 7: Missing from Gemini proposal entirely
- Alert-triggered profile switching (no mention)
- SMART trend alerting / predictive failure alerts (mentioned vaguely under "drive health")
- MQTT / Home Assistant integration (not mentioned)
- Notification channel expansion beyond webhook/email/push (not mentioned)
- Load-based sensor inputs (CPU %, GPU load) for fan curves (not mentioned)
- Config backup/restore (import/export) (not mentioned)
- C# dangerous-curve safety gate (was deferred from v2.2 and still missing)
- Moving average virtual sensor (key for stability, missing from proposed virtual sensor set)

## Part 4 — Refined Proposal

### Release 1: Parity + Operator UX (all items ship Python + C# together)

#### 1.1 C# Control Parity (3 items, 1 sprint)

**1.1a Hysteresis + ramp-rate in C# `ApplyCurvesAsync`**
Port `_apply_hysteresis()` and ramp-rate limiting from `fan_service.py:417-565` into C# `FanService.ApplyCurvesAsync`. Store deadband and ramp_rate_pct_per_sec in `SettingsStore` (the Python fields already exist; C# `SettingsController` already returns `fan_ramp_rate_pct_per_sec` from MEMORY audit, just not applied).

**1.1b Composite sensor inputs in C# FanCurve**
Add `SensorIds: List<string>` to `FanCurve` in `FanModels.cs`. Port `resolve_composite_temp()` into C# `FanService.ApplyCurvesAsync`. Initial support: MAX only (parity with Python). MIN/AVG/WEIGHTED are Release 2 via virtual sensors.

**1.1c PID mode in C# TemperatureTargetService**
Port `compute_pid_speed()` and `_PidState` to C#. Add `PidMode`, `PidKp`, `PidKi`, `PidKd` to `TemperatureTarget` model and DB schema. Wire into `TemperatureTargetService.Evaluate()`.

#### 1.2 C# Dangerous-Curve Safety Gate (deferred from v2.2)
Implement the `DANGER_TEMP_THRESHOLD` check equivalent from `curve_engine.py:78` in C# `FansController.cs`. Return 409 with `{ "detail": "curve is unsafe: ..." }` and log to audit table. This was explicitly deferred from v2.2.

#### 1.3 PID UI for Temperature Targets
`TemperatureTargetsPage.tsx` currently only renders proportional controls. Add a "Control mode" toggle (Proportional / PID). When PID is selected, show `Kp`, `Ki`, `Kd` sliders with sensible defaults and tooltips explaining each term. Show a live "estimated response" curve if possible. Match the Python behavior: `error = temp - target_temp_c`, integral anti-windup at ±200.

#### 1.4 Quiet Hours Frontend Page
Backend CRUD is complete. Add a `/quiet-hours` page to the frontend with:
- Weekly grid view (7 columns × time axis, like a calendar) as the primary view
- List view as secondary/fallback for small screens
- Create/edit rule form: day-of-week, start time, end time, profile selector (populated from `GET /api/profiles`)
- Toggle enable/disable per rule
- Validation: end_time > start_time; profile must exist
- Sidebar navigation entry

#### 1.5 Calibration Auto-Apply
In `FanTestPanel.tsx`, after a completed benchmark sweep, add an "Apply calibration" button alongside the existing results display. It calls `PUT /api/fans/{fanId}/settings` with `min_speed_pct` set to the measured stall+margin value and `zero_rpm_capable` set to the detected boolean. Show the current vs proposed values before confirming. Python and C# fan settings endpoints already exist.

#### 1.6 Config Backup / Restore
Add `GET /api/settings/export` → returns a JSON bundle containing: profiles, fan curves, temperature targets, alert rules, quiet hours rules, webhook config, email/push config, sensor labels. Add `POST /api/settings/import` that validates and applies the bundle atomically. Scope: both backends. This is table-stakes for any self-hosted tool — users need to back up and migrate configs.

### Release 2: Advanced Signals + Integrations

#### 2.1 Load-Based Sensor Inputs
Allow fan curves to be driven by CPU utilization %, GPU utilization %, and CPU package power (watts) where the hardware backend exposes them. These are first-class sensor readings: add `cpu_load`, `gpu_load`, `cpu_power` to `SensorType` enum. LHM exposes all three; `/metrics` from hwmon-based Linux backends may need kernel module checks. Update the curve editor sensor selector to group by type (Temperature / Load / Power).

This is the single most-requested feature from FanControl users — silent idle, loud only under actual load.

#### 2.2 Virtual Sensor CRUD

Promote virtual sensors from hardcoded composite logic (current MAX-only in `sensor_ids`) to a first-class named object stored in the DB:

```json
{
  "id": "vs_cpu_max",
  "name": "CPU Max (all cores)",
  "type": "max",           // max | min | avg | weighted | delta | moving_avg
  "source_ids": ["cpu_core_0", "cpu_core_1", ...],
  "weights": null,         // for weighted type only
  "window_seconds": null,  // for moving_avg type only
  "offset": 0.0            // constant offset applied after combination
}
```

- `delta`: `source_ids[0] - source_ids[1]` (coolant minus ambient, for AIO loop users)
- `moving_avg`: exponential moving average over N seconds (prevents thrashing on noisy sensors)
- Virtual sensors are resolved before curve evaluation and can be referenced by ID anywhere a real sensor ID is accepted (curves, temperature targets, alerts)

API: `GET/POST/PUT/DELETE /api/virtual-sensors`
C# and Python backends. Migration adds `virtual_sensors` table.

#### 2.3 Alert-Triggered Profile Switching

Extend the alert rule model with an optional `action` field:
```json
{
  "action": {
    "type": "switch_profile",
    "profile_id": "turbo_cooling",
    "revert_after_clear": true
  }
}
```

When an alert fires, the alert service calls `fan_service.apply_profile(action.profile_id)`. When the alert clears (condition no longer met), optionally revert to the previously active profile if `revert_after_clear` is true. This replaces the most common manual workflow: user watches temps, manually switches profile in a crisis. This is the "trigger-chain" model from iCUE/FanControl.

#### 2.4 SMART Trend Alerting

Currently drive health alerts are immediate-threshold only (via existing alert rules on sensor readings). Add trend-based alerting for SMART attributes:
- Reallocated sector count increasing by any amount within rolling 30-day window → fire `drive_degrading` alert
- Power-on hours crossing configurable threshold (e.g. 50,000h) → fire `drive_age` alert
- Wear percentage (`wear_percent_used`) crossing 80% / 95% thresholds

Store SMART snapshots in a dedicated `drive_health_snapshots` table (already partially in schema per MEMORY.md) with daily aggregation. Add trend computation to analytics report endpoint.

#### 2.5 Notification Channel Expansion

Current channels: webhook (HMAC-signed), email (SMTP), Web Push (VAPID).

Add:
- **ntfy.sh** — self-hosted push without app store dependency; one HTTP POST
- **Discord/Slack webhooks** — incoming webhook URL with message template; covers 80% of home-lab notification workflows
- **MQTT publish** — topic `drivechill/{machine_id}/alerts` and `drivechill/{machine_id}/sensors`; enables Home Assistant automation without a custom integration
  - Include auto-discovery payload on `homeassistant/sensor/drivechill_{machine_id}_{sensor_id}/config` for zero-config HA integration

All channels: same delivery pipeline and retry logic as existing webhook service. Add channel type enum to notification settings. Do not create separate services — extend the existing `WebhookService`/`webhook_service.py` dispatch pattern.

#### 2.6 Linux Fan Write Path (Production-Ready)

Current status: `lhm_direct_backend.py` is Windows-only (pythonnet + LibreHardwareMonitor DLL). Linux fan write is undocumented.

Implement:
- `hwmon_backend.py`: discovers `/sys/class/hwmon/hwmon*/pwm*` nodes; detects which ones are PWM-writable (`pwm*_enable`); writes via `echo N > /sys/class/hwmon/hwmonX/pwmY`
- Permission check at startup: warn if running without write access (`/sys` write requires root or `gpio` group on some distros)
- Capability flag `fan_write_supported: bool` on the health/capabilities endpoint
- UI: show a banner on the Fan Curves page if `fan_write_supported` is false, explaining why curves won't take effect

#### 2.7 USB Cooling Controller Support (liquidctl path)

Add a `liquidctl_backend.py` hardware backend that wraps liquidctl via subprocess (JSON output mode, `liquidctl --json status` / `liquidctl --json set ...`). Supported devices: NZXT Kraken, Corsair Commander Pro/Core, Aquacomputer D5 Next as initial targets.

Controller abstraction: `IHardwareBackend` already exists in C#; Python's `HardwareBackend` ABC in `base.py` is the equivalent. The liquidctl backend registers each device's fans as standard fan IDs and each pump/coolant temp as a sensor. No UI changes needed — devices appear as regular sensors and fans.

Document compatibility matrix: device → OS → tested status. This is the most-requested hardware addition from CoolerControl users who move to Windows.
